The Wisdom Vault

The Wisdom Vault is the factory's institutional memory. Every time a Commander successfully convinces the Critic in the War Room — achieving a CONVINCED verdict — the winning logic is automatically captured and converted into "Global Core" intelligence.

How Overrides Become Intelligence

The wisdom_accumulator.py module hooks directly into the Socratic Challenger's analyze_response() method via monkey-patching. When a verdict of CONVINCED is returned:

StepWhat HappensWhere It Lives
1. InterceptThe hooked analyze_response() detects verdict == "CONVINCED"wisdom_accumulator.py
2. ExtractChallenge ID, proposal text, the Commander's reasoning, and all analysis metrics are packagedIn-memory
3. ScoreThe combined score (original Critic + reasoning quality), strong/weak signal counts, and weakness coverage ratio are recordedIn-memory
4. PersistA JSON-line entry is appended to global_wisdom.db — one line per win, append-only, no data loss.system_core/global_wisdom.db
5. RecallFuture debates can query the vault: --query "scalability" returns all prior victories addressing that categoryCLI / API
6. Aggregate--stats reveals patterns: avg scores, strongest signal types, full-coverage win ratesCLI / API

This means the factory learns from its own debates. Over time, the Wisdom Vault reveals which argument structures work, which evidence types are most persuasive, and which weakness categories the Commander has mastered.

The Sanitization Protocol

Before any application folder is shared with a third party, investor, or external team, it must pass through the Deploy Shield — a multi-layered sanitization engine that ensures zero internal data leakage.

Three-Layer Sanitization

LayerWhat It DoesWhy It Matters
Layer 1: Structural PruningRemoves entire directories (__pycache__, .git, node_modules, socratic_logs, .audit_snapshots) and internal log filesEliminates debug artifacts, version history, and internal audit trails that contain operational fingerprints
Layer 2: Key StrippingRewrites every *_KEY, *_TOKEN, *_SECRET value in .env files to REDACTEDAPI keys and tokens are replaced — not deleted — so the file structure is preserved for the recipient to fill in their own credentials
Layer 3: PII MaskingRuns PIIMasker on all text-based source files (.py, .js, .json, .md, .html, .yaml)Catches stray file paths, IP addresses, email addresses, and hardcoded tokens embedded in source code

The output is a clean _deploy/ staging folder with a deploy_manifest.json that logs every file shipped, every redaction applied, and the total sanitization statistics. This manifest serves as a verifiable audit trail.

Budget Kill-Switch

The Kill-Switch is the factory's financial circuit breaker. It monitors the n8n execution rate and automatically disables all API keys if hourly spending exceeds a configurable threshold — preventing runaway costs from automated workflows.

Automated Protection Chain

📊 n8n_budget_guard.py → check_budget() → status == "critical" (≥ 90%) → kill_switch.check_and_panic() → 🔒 .env keys DISABLED
EventActionRecovery
Rate ≥ limit/hourBackup .env → .env.panic_backup, rewrite all key values to DISABLED_BY_KILLSWITCHpython kill_switch.py --restore
Manual panicpython kill_switch.py --panic — same as above but triggered by operatorSame restore command
Status checkpython kill_switch.py --status — shows ARMED (🟢) or TRIPPED (🔴)N/A (read-only)

The kill-switch is non-destructive: the original .env is always backed up first, and restore is a single command. All panic/restore events are logged to kill_switch_log.json (last 100 events retained).

Single App Maintenance

Child apps inherit from .system_core via symbolic links created by bootstrap_project.py. When updating a child app, follow these rules to avoid breaking the inheritance chain.

Safe Update Protocol

RuleDescription
DO Edit child-local filesAny file inside the child's own directory (e.g. MyApp/server.py) is safe to edit, replace, or delete
DO Update .system_core centrallyEdits to .system_core/*.py at the factory root automatically propagate to all child apps via symlinks — one fix benefits every project
DO Use bootstrap_project.pyAlways re-run bootstrap if adding new core modules — it will create any missing symlinks without disrupting existing ones
DON'T Delete symlink targetsNever delete a file in .system_core/ if any child still references it — check with dir /s *.symlink first
DON'T Replace symlinks with copiesReplacing a symlink with a local copy breaks the inheritance chain — that child will no longer receive upstream fixes
DON'T Move the project folderSymlinks use relative paths — moving the child folder to a different depth level breaks the .. resolution
