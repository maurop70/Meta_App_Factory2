██ SECTION 3 — BACKEND & API RULES [scope: backend]
3.1 — STRICT 502 GATEWAY FALLBACK (ANTI-SHATTER PROTOCOL)
All proxy routing functions forwarding traffic to backend microservices MUST
enclose async HTTP client requests (httpx.AsyncClient) in comprehensive try/except
blocks. On connection failure (timeout, refused, offline):

FORBIDDEN: allowing ASGI thread to shatter into raw 500
FORBIDDEN: swallowing the error with fail-open 200 OK
REQUIRED: explicit trap returning JSONResponse(status_code=502,
content={"error": "Gateway Unreachable", "detail": str(exception)})

3.2 — UNIFIED I/O SERIALIZATION ENVELOPE
All paginated collection routes MUST return this exact JSON envelope:
{"items": [...], "total": <int>, "limit": <int>, "offset": <int>}
Fragmented structures (arbitrary data keys, redundant "status": "success" strings)
are permanently forbidden.
3.3 — STRICT PAGINATION (NO UNBOUNDED QUERIES)
SELECT * FROM table is permanently forbidden. Every collection GET route natively
ingests and enforces limit and offset query parameters bound directly to the DB
execution layer. Frontend Axios must pass these parameters. Masking DOM bloat with
CSS overflow:hidden is a fatal architecture violation.
3.4 — EXPLICIT COLUMN ENUMERATION
Every SQL query must explicitly enumerate exact columns required. Fetching * or
unused columns to prevent memory inflation and DB-to-API serialization coupling
is forbidden.
3.5 — BACKEND ROUTING (NO CATCH-ALL, NO STATIC SERVING)
FastAPI routers are strict JSON I/O engines. Forbidden:

Catch-all fallback routes (/{full_path:path})
StaticFiles mounting to serve frontend assets
All routing must be explicitly defined. Unregistered paths fall through to strict
404 Not Found JSON exceptions.

3.6 — SAST BEFORE DISK I/O
All pre-flight static analysis, doctrine checks, and security validations must be
bound to the backend mutation endpoint (e.g., /api/v1/atomizer/mutate) prior to
disk I/O. Client-side React validation is strictly a visual aid — never the
primary security boundary. All pipelines must natively handle HTTP 403 exceptions.
3.7 — OBJECT-LEVEL AUTHORIZATION
All state-mutating DB operations must execute instance-level ownership checks
against the RS256 JWT payload (e.g., asserting assigned_tech == jwt_payload['sub'])
within the transaction block before mutation. RBAC alone without discrete
object-level ownership checks is a fatal authorization bypass.
3.8 — NGINX TRAILING SLASH DOCTRINE
All Nginx blocks proxying to internal microservices MUST enforce trailing slashes
on both location boundary and proxy_pass directive:
location /auth/ { proxy_pass http://127.0.0.1:9000/; }
Omitting trailing slash causes silent URI stripping and fatal backend routing errors.

██ SECTION 4 — FRONTEND & UI RULES [scope: frontend]
4.1 — NATURAL LANGUAGE GATEWAY ISOLATION
Frontend components (e.g., NaturalLanguageGateway.jsx) must strictly isolate
biological user input from system-generated UI status logs. Concatenating [SYSTEM]
telemetry messages into the outbound chatLog payload causes recursive logic
poisoning and fatal routing parse failures at the ASGI boundary.
4.2 — AXIOS DOUBLE-PREFIX PROHIBITION
When an Axios instance is pre-configured with baseURL (e.g., /api), all downstream
component calls MUST omit the root prefix:
CORRECT:   api.get('/resource')
FORBIDDEN: api.get('/api/resource')  ← causes fatal /api/api/resource routing
4.3 — REACT PORTAL REQUIREMENT
All enterprise UI modals, blocking overlays, and floating context menus MUST be
physically ejected to the root DOM using ReactDOM.createPortal(element, document.body).
4.4 — REACT PORTAL MUTATION LOCKOUT
Any React Portal executing async DB or state mutations MUST forcefully disable all
closure vectors (Escape key listeners, backdrop click handlers, close buttons) during
both the loading phase AND the success resolution phase. Structural teardown remains
locked until all post-mutation callbacks (delays, parent state refreshes) have fully
executed. Prevents state-cancellation memory leaks.
4.5 — DESTRUCTIVE ACTIONS IN DETAIL MODAL
Destructive or state-advancing actuations (Approve/Reject) are forbidden as inline
table elements. They MUST be isolated in a constrained Detail Modal to force the
operator to ingest contextual payload before executing.
4.6 — RESPONSIVE TABLE REFLOW (>5 COLUMNS)
Enterprise data tables exceeding 5 columns MUST implement a data-label CSS reflow
matrix. Do not build parallel mobile components. Dismantle <table> into stacked,
touch-friendly Key-Value cards on viewports < 900px.
4.7 — CATEGORICAL DATA — NO FREE-TEXT DOM INPUTS
All bounded enterprise domains (Categories, Departments, Status Enums) MUST use
Foreign Key relational bindings backed by paginated backend lookup routes. Free-text
DOM inputs for categorical data are forbidden. Backend engine synthesizes all PKs;
frontend payload must never include a PK on ingestion actuations.

██ SECTION 7 — INFRASTRUCTURE & DEPLOYMENT [scope: infra]
7.1 — PRODUCTION DATABASE EXCLUSION (CI/CD)
All production databases using physical filesystems (SQLite/WAL) MUST enforce
.db exclusion in CI/CD deployment archiving. This prevents temporal state
injection and irreversible WAL matrix corruption. Initializing a blank remote
cluster requires isolated DDL schematization scripts followed by DML base IAM
seed injection. Booting an API Gateway against a void filesystem triggers fatal
schema failures.
7.2 — DEPLOYMENT: NO CLOUD PIPELINES FOR BOUNDED DOMAINS
For bounded enterprise domains with physical SQLite/WAL matrices: Vercel, Render,
and GitHub Actions are permanently forbidden. All CD is orchestrated via standalone
native OS Python applications using subprocess.Popen to execute direct scp/ssh
transmissions to persistent bare-metal or VPS (DigitalOcean) hosts.
7.3 — CONCURRENT DAEMON RESTART
When executing remote CI/CD transmissions or code mutations to multi-service
architecture: ALL dependent daemon processes MUST be cycled concurrently in the
same execution block (e.g., systemctl restart erp-backend erp-auth). Restarting
the primary daemon while neglecting the Identity Gateway causes fatal memory state
desynchronization.
7.4 — SOCKET-BASED HEALTH AS GROUND TRUTH
Autonomous dashboards (Socratic App Registry, System Map) are forbidden from
determining agent state by polling internal Orchestrator memory ledgers or PIDs.
All registry state engines MUST verify agent life by polling the physical network
socket (e.g., GET http://127.0.0.1:<PORT>/health). Socket binding is the absolute
source of truth. Prevents split-brain state fractures when nodes are manually
booted via OS terminal.
7.5 — ROLLBACK RECIPE BEFORE EVERY DEPLOY (added 2026-06-11)
Every production deployment MUST record a rollback recipe BEFORE mutating the
target: {git SHA deployed, pre-migration DB backup path, services restarted,
timestamp} written to logs/deploy_recipes.jsonl. The post-deploy health probe is
part of the deploy, not an afterthought: non-200 within the probe window triggers
the rollback recipe automatically. A deploy without a recorded rollback recipe
is an incomplete deploy.

  2. git_operation(add, paths=[<specific files, never ".">])
  3. git_operation(commit, message="<type>(<scope>): <description>")
  4. git_operation(push, branch="main")
  5. execute_shell("python deploy_maf.py")    ← MAF, cwd: MAF root

  6. execute_remote_shell — systemctl status check
  7. execute_remote_shell — curl health endpoint → expect 200

██ SECTION 13 — ARCHITECT DISCIPLINE (added 2026-06-11) [scope: core]
Ported from the proven operator↔Architect collaboration rules.

13.1 — CRITICAL PLAN REVIEW BEFORE DISPATCH
No mandate is dispatched verbatim — not even the Architect's own. Before
dispatch, review the plan against the live codebase for: route/name collisions,
schema mismatches, RBAC inconsistencies, missing pieces implied by the
verification section, security/concurrency risks, and simpler alternatives.
Apply uncontroversial corrections (noting them in the mandate); raise
scope-changing concerns to the operator. Past defects caught this way: route
collisions with dead legacy code, missing UI columns required by the plan's own
verification, wrong credentials in test steps, deploy scripts missing
pre-migration backups.

13.2 — PROACTIVE RULE CREATION (POSTMORTEM ENGINE)
Every ERROR or ESCALATE run outcome triggers a postmortem draft: root cause,
what rule would have prevented it, proposed rule text. The draft is written to
rules/pending_rules.jsonl with provenance {trace_id, date, root_cause}. The
OPERATOR approves before any rule enters this file. Near-misses count: if a
mistake was caught before damage, codify the prevention anyway. Check for an
existing rule covering the failure before proposing a duplicate. Rules that
have not fired in 6 months are flagged for deprecation review — the rulebook
must not only grow.

13.3 — VERIFICATION CONTRACTS ARE IMMUTABLE SIGNALS
Established test suites and their pass counts (rules/verification_contracts.json)
are a contract. The Architect NEVER mutates an existing suite's expectations to
accommodate new work — new features get NEW suites, and the contract registry
gains a new entry. A changed count must always mean someone deliberately changed
the contract, flagged explicitly to the operator. Ledgers must report results
per suite with expected totals so drift is visible at a glance.

13.4 — EPISODIC RECALL BEFORE DIAGNOSIS
Before diagnosing a failure, query episodic memory (logs/episodes.jsonl) for
similar past mandates. A failure that matches a past episode gets the past
resolution offered as the first hypothesis — many "bugs" are environmental
repeats (stale process needing restart, unmounted drive, expired session).

██ SECTION 14 — EXECUTOR DISCIPLINE (added 2026-06-11) [scope: core]
Ported from the proven operator↔Architect collaboration rules.

14.1 — STRUCTURED LEDGER CONTRACT
Every ledger MUST end with a machine-readable block:
```
LEDGER_JSON: {"status": "COMPLETE|ITERATE|ERROR|ESCALATE",
 "summary": "<one line>", "files_changed": [...],
 "tests_run": [{"suite": "...", "passed": N, "failed": N}],
 "next_step": "<or null>", "needs_human": "<reason or null>"}
```
status reflects reality: failing tests = ERROR or ITERATE, never COMPLETE.
Tier 3 territory encountered mid-mandate = ESCALATE with needs_human set.
A ledger without this block is treated as low-confidence and re-verified.

14.2 — LIVE-PROBE DISCIPLINE
When verifying against a RUNNING service (not an isolated test copy), every
probe is potentially state-changing. Prefer read-only requests (GET, or
auth-failure probes) to confirm behavior. If a state-changing probe is
unavoidable: know beforehand exactly what artifact it creates, delete it
immediately after (matching its exact content, not a broad filter), and
disclose both the side effect and the cleanup in the ledger. Before declaring
"restart required", test whether the service hot-reloads.

14.3 — MIGRATION BACKUP RETENTION
Back up before any ALTER/DROP/table-rebuild; print the backup path so it is in
the ledger. Never bulk-delete backups with wildcards in the same session that
created them. Only remove migration backups when the operator explicitly says
the system is verified healthy. (ENFORCED: fs_wire blocks deletion of backup-
patterned files; deploy_erp.py takes pre-migration backups automatically.)

14.4 — SELECTIVE STAGING ONLY
The repo is shared with other agents whose half-finished work is always in the
tree. Stage only files you authored, by explicit path. Never stage:
__pycache__/.pyc, logs/, dist/ builds, .db files, screenshots, other agents'
files. (ENFORCED: git_wire blocks add ".", "-A", "--all".)

14.5 — PUSH IS NOT DEPLOY
Push to GitHub does NOT deploy. Deployment is a separate Tier 3 act via the
dedicated deploy scripts. Never run a deploy script because a push succeeded.

14.6 — SAME-SESSION SEAL OF VERIFIED WORK
Work that passes its verification contract is committed and pushed in the same
session (rule 2.4). Uncommitted verified work is invisible to other agents and
unprotected against loss.

██ SECTION 15 — RISK-TIERED AUTONOMY (added 2026-06-11) [scope: core]
Replaces keyword-based Section 11 gating in the loop engine. The tier is
classified from the INSTRUCTION text and the mandate's declared scope — never
from the full rule-injected prompt (which always contains gate words).

TIER 0 — READ-ONLY DIAGNOSIS. Reads, status, logs, health probes, telemetry.
  Autonomy: full. No gate.
TIER 1 — LOCAL WORKSPACE MUTATION. File edits, test runs, local builds,
  local service restarts. Autonomy: full, Auditor verifies the ledger.
TIER 2 — VERSION CONTROL. add (explicit paths) / commit / push to dev or main.
  Autonomy: full, structured ledger + Auditor verification required.
TIER 3 — IRREVERSIBLE OR OUTWARD-FACING. Production deploys, schema
  migrations on live DBs, data deletion/overwrite, merges of dev→main,
  new Child Apps, rule changes, business-logic/UX decisions.
  Autonomy: NONE. Operator approval required (Section 11). The autonomy
  trigger's self-healing remediation is capped at Tier 3-R: service
  restart and rollback-recipe execution on the approved hosts only —
  never new-code deploys.

The Executor declares the tier it believes a mandate requires in LEDGER_JSON
(needs_human) when it encounters higher-tier territory mid-execution. Tier
violations are fatal: the Auditor cross-checks audit logs (git_wire, ssh_wire)
against the mandate's authorized tier.
